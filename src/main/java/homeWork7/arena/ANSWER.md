### 1. Почему `heroesCreated` должно быть статическим?
Потому что это свойство принадлежит не конкретному персонажу, а всему классу `Hero` в целом. Нам нужно считать общее количество созданных героев в игре. Если бы поле было нестатическим, у каждого героя был бы свой собственный счетчик, равный 1.

### 2. Почему `name`, `health` и `level` не должны быть статическими?
Это индивидуальные характеристики каждого персонажа. Если сделать их статическими, то у всех героев в игре будет абсолютно одинаковое имя, одинаковый уровень и общее здоровье на всех (нанесли урон одному — здоровье уменьшилось у всех).

### 3. Почему `printHeroesCreated()` можно вызвать без создания объекта?
Потому что этот метод помечен модификатором `static`. Статические методы привязаны к самому классу, а не к его экземплярам. Они загружаются в память вместе с классом, поэтому их можно вызывать напрямую через имя класса: `Hero.printHeroesCreated()`.

### 4. Можно ли из статического метода напрямую обратиться к нестатическому полю `name`? Почему?
**Нет, нельзя.** Статический метод существует на уровне класса и может быть вызван, когда ни одного объекта еще не создано. Нестатическое поле `name` появляется только тогда, когда мы пишем `new Hero(...)`. Статический метод просто "не знает", чьё именно имя ему нужно взять.

### 5. Почему проект перестанет компилироваться, если пометить класс `Hero` модификатором `final`?
Модификатор `final` для класса запрещает наследование от него. Классы `Knight`, `Archer` и `Mage` используют ключевое слово `extends Hero`. Если `Hero` станет `final`, компилятор выдаст ошибку, так как порождение наследников будет заблокировано.

### 6. Почему переменные имеют тип `Hero`, но при этом вызываются версии метода `attack()` из конкретных наследников?
Это фундаментальное проявление **полиморфизма** в Java (а именно — динамическое/позднее связывание). В цикле тип ссылки — `Hero` (это контракт, который гарантирует, что метод `attack()` существует), но виртуальная машина Java (JVM) в момент выполнения программы смотрит на *реальный тип объекта* в куче (Heap) и вызывает именно его переопределенную версию.

### 7. Почему переменная `knight` имеет модификатор `final`, но ее поля могут быть изменены?
Модификатор `final` для ссылочной переменной означает, что **нельзя изменить саму ссылку** (то есть нельзя заставить переменную `knight` указывать на другой, новый объект через `new`). Однако `final` никак не защищает внутреннее состояние самого объекта. До тех пор, пока внутренние поля (например, `armor`) не помечены как `final`, мы можем свободно их менять с помощью сеттеров или методов. Изменяется "содержимое коробки", но сама "коробка" остается прежней.
